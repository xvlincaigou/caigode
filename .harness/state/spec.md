# Product Spec
- 问题定义：
  - 当前仓库只有 harness 模板，缺少一个独立的 `coding agent CLI` 产品实现。
  - 目标产品不是扩展或改写 [codex.sh](/Users/xvlin/workspace/caigode/codex.sh)，而是新增一套以 Python 为主运行时的代理程序。
  - 产品需要支持交互式多轮会话、工作区内文件读写、本地命令执行，以及工程交付材料输出。
  - 模型接入方式已确定为直连 OpenAI 兼容 API，且必须允许自定义 `model` 与 `base_url`。
  - Python 项目管理与依赖管理工具已确定为 `uv`。
- 业务目标：
  - 提供一个新的多子命令 CLI，作为本地 coding agent 的正式产品入口。
  - 支持用户在目标代码仓库中进行交互式任务澄清、文件修改、命令验证与交付总结。
  - 通过文件状态持久化会话、计划、执行记录和交付产物，保证本地可恢复与可审计。
- 成功指标（可量化）：
  - CLI 至少提供 `chat`、`run`、`status`、`review` 4 个子命令，且 `--help` 均返回退出码 `0`。
  - 配置层同时支持命令行参数与环境变量设置 `model`、`base_url`、`api_key`，并有自动化测试覆盖。
  - 在受控测试夹具中，代理可完成一次最小闭环：读取任务、修改文件、执行本地验证命令、写出会话状态与交付摘要。
  - 会话状态与交付产物均写入文件系统，并可通过 `status`/`review` 子命令读取。
  - 最小自动化测试可在本地一次性运行通过。
  - 项目可通过 `uv sync` 完成依赖安装，并通过 `uv run` 执行 CLI 与测试命令。
- 约束（时间/人力/技术/合规）：
  - 架构采用 A1 三层：接口层 / 领域层 / 基础设施层。
  - 模块边界纳入 B1+B2+B3+B4+B5。
  - 状态策略采用 C1：文件状态为主。
  - MVP 边界采用 D2：本地 + 最小自动化测试。
  - 交互模式采用 E2：交互式会话（REPL/聊天式多轮）。
  - 执行权限边界采用 F2：工作区读写 + 本地命令执行；外部网络仅允许访问显式配置的 OpenAI 兼容模型端点。
  - 能力闭包采用 G3：工程交付闭环，输出本地交付材料，不默认负责远程发布。
  - 实现运行时采用 H2：Python 为主。
  - Python 项目管理采用 `uv`，不使用 `poetry`、`pipenv` 或独立 `requirements.txt` 作为主入口。
  - CLI 入口契约采用 I2：多子命令 CLI。
  - 模型集成采用 J2：直连 OpenAI 兼容 API，并支持自定义模型名与 URL。
- 非目标（明确不做）：
  - 不改造现有 harness 的执行框架，不把产品入口绑死到 `codex.sh`。
  - 不在 MVP 中实现远程 PR 创建、CI 执行托管、浏览器自动化或多用户协作。
  - 不默认支持任意第三方工具生态、插件市场或分布式任务调度。
  - 不在 MVP 中引入数据库、消息队列或长期后台 Worker。
- 用户场景与主流程：
  - 场景 1：用户在目标仓库执行 `caigode chat`，输入需求，与代理进行多轮澄清和执行。
  - 场景 2：代理分析工作区，生成计划，读取/修改文件，执行本地验证命令，并持续将状态写入本地文件。
  - 场景 3：用户执行 `caigode status` 查看会话进度、待办、验证结果和最近产物。
  - 场景 4：用户执行 `caigode review` 生成工程交付摘要，例如变更说明、验证摘要、提交说明草稿。
  - 场景 5：用户执行 `caigode run` 以非交互方式消费输入任务并产出同类状态与交付文件。
- MVP 验收标准：
  - `pyproject.toml`、`src/caigode/`、`tests/` 建立为一个独立 Python 项目，且不依赖改写 `codex.sh` 才能运行。
  - `uv sync` 可成功解析并安装项目依赖。
  - `uv run python -m caigode.cli --help` 与 `uv run python -m caigode.cli chat --help`、`run --help`、`status --help`、`review --help` 全部返回退出码 `0`。
  - 代理可从命令行或环境变量读取 `OPENAI_API_KEY`、`OPENAI_BASE_URL`、`OPENAI_MODEL` 等配置，并在缺参时给出明确错误。
  - 在测试夹具仓库中，至少一条自动化测试通过 `uv run pytest` 覆盖“模型响应 -> 文件改动 -> 本地命令验证 -> 产出 review 文件”的闭环流程。
  - 会话状态、操作日志、验证结果、交付摘要均可在本地文件系统中定位并复读。
